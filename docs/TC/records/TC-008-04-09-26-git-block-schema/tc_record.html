Overview

Summary

Extend the tc_record schema with an optional `git` block capturing commit SHAs, branch info, remotes, and PR metadata. Foundation for all subsequent git integration work (TC-009 through TC-016).

Motivation

The TC system was entirely VCS-agnostic, with /tc retro --from-git as the sole git touchpoint. To enable commit-level linking, auto-link hooks, PR enrichment, and git-aware dashboards, records must carry git metadata. Designed as an optional top-level object so the 7 existing TCs remain valid with zero migration.

Detailed Design

Added top-level `git` property to tc_record.schema.json: { repo_root, initial_branch, commits[], remotes[], release_tags[] }. Each commit entry is append-only with {sha, short_sha, author, authored_date, subject, branch, parent_count, files_changed[], linked_at, link_source}. Each remote has optional nested pr object with {number, url, state, merged_sha, review_decision, last_synced}. Validator enforces SHA regex ^[0-9a-f]{7,40}$, deduplicates SHAs within a record, and validates provider enum (github|gitlab|bitbucket|other) and PR state enum (open|merged|closed|draft). Backward compatible: absent/null git block passes validation.

Decisions Made
Git block is top-level optional object, not nested inside metadata
Top-level placement lets /tc git status and HTML generators find it without digging, and matches the precedent of files_affected[] and session_context being siblings of metadata.
2026-04-09 10:10 UTC-04:00
Enforce SHA uniqueness within git.commits[] at validator level
Prevents accidental double-linking from auto-hook + manual /tc link. Duplicate detection is cheap (set) and the error message is clearer than catching it downstream.
2026-04-09 10:20 UTC-04:00
PR block is nested under remotes[], not top-level git.pr
A TC could theoretically span forks or mirror remotes. Nesting PR under the remote keeps the data model correct if that ever happens, and degrades cleanly to a single origin remote in the common case.
